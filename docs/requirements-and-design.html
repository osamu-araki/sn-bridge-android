1. 背景と現状の課題

1.1 現在のアーキテクチャ

ChromeBridge は Android 端末上で NanoHTTPd + WebView によるヘッドレスブラウジングを提供し、Cloudflare Tunnel 経由で外部からアクセス可能にするアプリケーションです。

項目現在の実装課題
処理モデル LinkedBlockingQueue + 1ワーカースレッド 直列処理のため、1リクエスト最大43秒間後続が待機
統計情報 GET /status で pending_requests, uptime のみ 累計処理件数・通信量・エラー率が把握不可
ログ インメモリ CopyOnWriteArrayList（最大100件） 再起動で消失。日別・月別の集計不可
並列数 固定: 1（MAX_QUEUE_SIZE = 10） 設定変更不可。スループット向上の余地なし
タイムアウト クライアント指定値をそのまま使用（上限なし） 悪意あるリクエストで長時間スレッド占有が可能

1.2 現在のタイムアウト構造

箇所現在の値コード問題
WebView 読み込み待ち timeout パラメータ（デフォルト 30秒） HeadlessWebViewFetcher:104 クライアントが任意の値を指定可能（上限なし）
ページ読み込み後の待機 wait パラメータ（デフォルト 3秒） HeadlessWebViewFetcher:83 同上。上限バリデーションなし
サーバー側の結果待ち timeout + 10秒（デフォルト 40秒） BridgeHttpServer:182 timeout に連動して無制限に伸びる
1リクエスト最大所要時間 wait + timeout + 10秒 = 43秒（デフォルト）、上限なし

1.3 解決すべき問題

  1. 運用可視性の欠如 - 処理件数・通信量・エラー率を日別/月別で確認できない
  2. スループットの制限 - 直列処理が性能ボトルネック。複数WebViewで並列化したい
  3. 設定の柔軟性不足 - 並列数・キューサイズ・タイムアウトを動的に変更できない
  4. タイムアウトの安全性 - クライアントが極端な値を送信してリソースを占有できる

3.2 表示要件

日別表示

  • 直近30日分のデータをリスト形式で表示
  • 日付、リクエスト数、成功率(%)、通信量(KB/MB自動切替)、平均処理時間
  • 当日のデータはリアルタイム更新（5秒間隔）

月別表示

  • 直近12ヶ月分のデータをリスト形式 + 棒グラフで表示
  • 月、リクエスト数、成功率(%)、通信量(MB/GB自動切替)、平均処理時間

サマリーカード（ダッシュボード上部）

  • 今日の処理件数 / 成功率
  • 今日の通信量
  • 今月の処理件数 / 通信量
  • 現在の並列実行数 / 設定上限

3.3 エラー分類

エラーコード分類説明
TIMEOUT処理系WebView の読み込みがタイムアウト
FETCH_FAILED処理系WebView がページ取得に失敗（ネットワーク等）

PARSE_ERROR処理系JS実行結果のパースに失敗

4. 機能要件 - 並列処理

4.1 並列処理モデル

現在 (v1.x)

  • WebView: 1インスタンス
  • ワーカースレッド: 1
  • キューサイズ: 10（固定）
  • 最大スループット: ~1.5 req/min

提案 (v2.0)

  • WebView: N インスタンス（プール）
  • ワーカースレッド: N（設定可能）
  • キューサイズ: M（設定可能）
  • 最大スループット: ~N × 1.5 req/min

4.2 設定項目

設定名型デフォルト範囲説明
並列数 (concurrency)Int21 ~ 8同時に処理可能なリクエスト数（= WebView プールサイズ）
キューサイズ (queueSize)Int205 ~ 100待機キューの最大長。メモリ上の参照のみでリソース消費は軽微
最大タイムアウト (maxTimeout)Int (秒)6010 ~ 120クライアントの timeout パラメータの上限
最大wait (maxWait)Int (秒)101 ~ 30クライアントの wait パラメータの上限
WebView のメインスレッド制約: Android の WebView は UI スレッド（メインスレッド）でのみ操作可能。複数 WebView インスタンスを生成しても、loadUrl() や evaluateJavascript() の呼び出しはすべてメインスレッドにディスパッチされる。これは Android フレームワークの制約であり回避不可。

ただし、各 WebView は独立してページ読み込みを実行するため、メインスレッドへのポストは瞬時に完了し、実際のネットワーク通信・レンダリングは WebView 内部のバックグラウンドスレッドで並行して行われる。よって 複数 WebView の同時ロードによる実質的な並列処理は可能。

4.3 並列数と端末スペックの目安

重要: 並列数の選択は端末スペックに大きく依存します。 下表を参考に、運用端末に適した値を設定してください。不適切な並列数はOOMクラッシュやANRの原因になります。
並列数推定メモリ増加推奨端末スペック注意事項
1 +50~100 MB 全端末で安全 v1.x 互換モード。確実に動作する
2 (デフォルト) +100~200 MB RAM 4GB 以上 大半の端末で安定動作。推奨設定
3~4 +200~400 MB RAM 6GB 以上 ミドルレンジ以上。他アプリとの競合に注意
5~6 +400~600 MB RAM 8GB 以上 ハイエンド端末向け。専用機としての運用を推奨
7~8 +600~800 MB RAM 10GB 以上（ハイエンド） Galaxy S / Pixel Pro クラスの最新端末が必要。
端末を ChromeBridge 専用機として運用すること。
バックグラウンドアプリは最小限に抑える

4.4 並列処理の振る舞い

  1. アプリ起動時に設定値分の WebView インスタンスをプールに生成
  2. リクエスト到着時、プールから空き WebView を取得（Semaphore で制御）
  3. 空きがなければキューで待機。キューも満杯なら 503 を返却
  4. フェッチ完了後、WebView をプールに返却
  5. 並列数の変更はサーバー再起動時に反映（ホットリロード不要）

4.5 安全性の考慮

  • 並列数の上限を 8 に制限（ハードリミット。コード上で固定）
  • 端末メモリ 4GB 未満で並列数 3以上を設定した場合、UIに警告を表示
  • onLowMemory() コールバックで WebView プールサイズを自動縮小（最低1まで）
  • WebView プール内のインスタンスが応答しない場合、タイムアウト後に破棄・再生成
  • キューサイズ最大100でもメモリ影響は軽微（関数参照のみ保持、WebView リソースは消費しない）

5. 機能要件 - タイムアウト制御

5.1 サーバー側クランプ

クライアントから送信された timeout と wait の値を、サーバー設定の上限値で制限（クランプ）する。

// クライアント指定値をサーバー上限でクランプ
val effectiveTimeout = min(request.timeout, settings.maxTimeout)  // max 60s
val effectiveWait    = min(request.wait, settings.maxWait)        // max 10s

// 1リクエストの最大所要時間
// = effectiveWait + effectiveTimeout + バッファ(10s)
// = 10 + 60 + 10 = 最大80秒（デフォルト設定時）

5.2 タイムアウト構造（変更後）

箇所v1.x（現在）v2.0（変更後）
WebView 読み込み待ち timeout（上限なし） min(timeout, maxTimeout) デフォルト上限 60秒
ページ読み込み後の待機 wait（上限なし） min(wait, maxWait) デフォルト上限 10秒
サーバー側の結果待ち timeout + 10秒 effectiveTimeout + effectiveWait + 10秒
1リクエスト最大所要時間 無制限 最大 80秒（デフォルト設定時）

5.3 クランプ適用時のレスポンス

クライアント指定値がクランプされた場合、レスポンスに clamped フィールドを追加して通知する。

{
  "ok": true,
  "clamped": {
    "timeout": { "requested": 300, "applied": 60 },
    "wait": { "requested": 30, "applied": 10 }
  },
  "text": "..."
}

8.1 SQLite スキーマ

request_log テーブル（個別リクエスト記録）

CREATE TABLE request_log (
    id             INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp      INTEGER NOT NULL,     -- Unix epoch millis
    date_key       TEXT    NOT NULL,     -- "2026-03-11"
    month_key      TEXT    NOT NULL,     -- "2026-03"
    url            TEXT    NOT NULL,
    success        INTEGER NOT NULL,     -- 1=成功, 0=失敗
    error_code     TEXT,                    -- TIMEOUT, FETCH_FAILED, etc.
    duration_ms    INTEGER NOT NULL,     -- 処理時間(ms)
    response_bytes INTEGER NOT NULL,     -- レスポンスサイズ(bytes)

CREATE INDEX idx_date ON request_log(date_key);
CREATE INDEX idx_month ON request_log(month_key);

11.3 リスクと対策

リスク影響対策
複数 WebView のメモリ消費 OOM クラッシュ 上限8。設定UI でメモリ警告を表示。onLowMemory() でプール自動縮小（最低1）。起動時にメモリチェック
WebView メインスレッド競合 ANR loadUrl/evaluateJs のメインスレッドポストは瞬時に完了。実際の処理は WebView 内部スレッド
SQLite の書き込み負荷 処理遅延 メモリバッファ + 5秒間隔バッチ書き込みで I/O を最小化
クライアントによるリソース占有 DoS timeout/wait をサーバー側でクランプ。レスポンスに clamped 情報を含めて透明性確保
統計データの整合性 表示値のずれ 日次バッチで request_log から daily_stats を再計算。月次も同様
設定変更時の不整合 サーバー動作不安定 並列数変更はサーバー再起動時のみ反映。タイムアウト変更は即座に反映
実装完了（2026-03-11）: 全4フェーズの実装が完了。Phase 1（データ基盤） → Phase 2（並列処理 + タイムアウト制御） → Phase 3（3タブ UI + /stats API） → Phase 4（クリーンアップ + onTrimMemory + readBody修正 + 認証強化）の順で段階的に実装。ビルド・実機テスト後にリリース可能な状態。
